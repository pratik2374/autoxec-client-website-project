# AutoXec Architecture & System Design Blueprint

This document outlines the production-grade architecture, scalability strategies, cost-efficiency mechanisms, and CI/CD operations for the AutoXec platform. 

---

## 1. High-Level Decoupled Architecture

AutoXec follows a **Headless / Decoupled Application Architecture**. We do not rely on traditional monolithic servers (like WordPress/PHP).

*   **Content Layer (Backend)**: Sanity CMS. Acts purely as an API delivering structured JSON.
*   **Application Layer (Frontend)**: React 19 SPA (Single Page Application) built with Vite.
*   **Delivery Layer (Edge)**: Hosted via CDN (AWS CloudFront / Vercel), ensuring millisecond response times globally.

### Fallback & Resilience Strategy
We do not use hardcoded "stub" data. Instead:
1.  On successful load, `fetchSite.ts` serializes the entire site context into the browser's `localStorage` as `autoxec_content_cache`.
2.  If the Sanity Content Lake goes down, or the user enters a sub-optimal network zone (e.g., subway), the app gracefully reads from the `localStorage` cache.
3.  If this is their first visit *and* the API is down, a production-grade **Error Boundary Overlay** is triggered, informing the user of a temporary system outage.

## 2. Global Scalability & Hosting (AWS Strategy)

Because AutoXec is a Client-Side Rendered (CSR) application, traditional constraints like compute load (CPU) or database connection limits do not apply to the Frontend.

### Target Deployment: AWS Native
We will deploy the application directly to AWS to maintain high availability and security.

*   **Storage**: Amazon S3 (Standard Storage). The `dist/` folder generated by `npm run build` is placed directly in S3. S3 can handle infinite concurrent requests for static files.
*   **CDN**: Amazon CloudFront. CloudFront caches the S3 files at Edge Locations across India (Mumbai, Delhi, Chennai) and globally. This reduces S3 egress costs and drops Latency for users to <50ms.
*   **Routing**: AWS Route53 will handle DNS, pointing `autoxec.in` to the CloudFront distribution.
*   **CMS Scalability**: Sanity scales automatically. They utilize Google Cloud Platform's global edge network to cache API responses. Our queries use `useCdn: true` in the production client config to leverage this.

## 3. Search Implementation Strategy

### Current Status: GROQ-Based In-Memory Search
Currently, search is handled natively via GROQ. When a user types a query, it filters the `articles` array within the `SiteDataContext` or pings the Sanity API directly with a `match` filter.
*   **Pros**: $0 overhead. It's completely free and natively integrated.
*   **Cons**: Lacks typo-tolerance and complex weighting (e.g., prioritizing title over body text).

### Future Upgrade: Algolia Search
As the community scales past 5,000+ articles, we will integrate **Algolia**.
*   **Architecture**: We will set up a **Sanity Webhook**. Every time an article is published or deleted in Sanity, the webhook fires to an AWS Lambda function, which updates the Algolia index.
*   **Cost Efficiency**: Algolia is priced per 1,000 requests. To protect margins, we will heavily debouce user input on the frontend (e.g., waiting 400ms after the user stops typing) before firing the search request to Algolia.

## 4. User Management & Expanding the "Community"

With the introduction of the *Field Researchers Community*, we require authentication and user mapping to capture data efficiently. 

### Database & Auth Stack
We have chosen **Firebase** (currently integrated into `package.json`).
*   **Authentication**: Firebase Auth will handle OTP-based Mobile Login and Google OAuth. This is extremely cost-effective (free up to 50k MAUs) and highly secure.
*   **User Profiles**: Firestore Database will house the `users` collection.
    *   *Schema fields*: `name`, `mobile`, `city`, `primaryVehicle`, `intelligencePoints` (IP).
*   **Sync Logic**: While CMS data lives in Sanity, User Generated Content (comments, forum posts, upvotes) will live in Firestore. This prevents our Sanity instance from running up costly mutation limits for high-frequency user actions.

## 5. CI/CD Operations (Continuous Integration / Continuous Deployment)

We will use **GitHub Actions** to automate our deployment pipeline entirely, ensuring nobody is manually uploading files to AWS.

### The Pipeline Workflow (`.github/workflows/deploy.yml`)
1.  **Trigger**: Code is merged into the `main` branch.
2.  **Build**: A clean Ubuntu runner spins up, runs `npm install`, then executes `npm run build` (invoking Vite and TypeScript compilation).
3.  **Test**: Running any frontend integration tests (Playwright, Jest).
4.  **Sync to S3**: The pipeline uses `aws-cli` to run `aws s3 sync ./dist s3://autoxec-production-bucket --delete`. This completely overwrites the old codebase with the new one.
5.  **Invalidate CDN**: A final step runs `aws cloudfront create-invalidation --distribution-id $PROD_ID --paths "/*"`. This forces edge servers worldwide to drop their old versions of our JS/CSS files and pull the new ones from S3 instantly.

This automated pipeline guarantees zero downtime during deployments.
